# Skiller — Skill 设计教练

> 不只是一键生成 Skill，而是帮你判断值不值得做、怎么做对、怎么持续维护。

## 触发条件

- 用户要求创建、编写、生成、设计一个新的 Skill
- 用户提到"写个 Skill"、"建个 Skill"、"Skill 化某个任务"、"把流程沉淀成 Skill"
- 用户想评估、优化或改造已有 Skill（触发迭代维护流程）
- 用户遇到路由冲突、Skill 不稳定、误触发等问题，需要诊断
- 用户想知道某个 Skill 是否该退役

## 不适用

- 用户只是想了解 Skill 的概念（直接解释即可，不需要加载完整 Skill）
- 用户想搜索或安装社区现成 Skill（引导至 skills.sh 或对应平台的 Skill 市场）

## 核心原则（写进每个 Skill 的 DNA）

在生成或评估任何 Skill 之前，必须牢记这些原则：

### 原则 1：写动作，不写原则

每一条指令都必须是 Agent 可以直接执行的动作，而不是抽象的原则性描述。

| 禁止写法 | 正确写法 |
|---|---|
| 保证内容高质量 | 先提炼读者、目标和主线；标题要具体；连续纯文本超过 3 段时改成表格、列表或 callout |
| 充分利用工具 | 遇到某类链接时先用对应命令读取；只改局部时优先按 section 读取 |
| 必要时询问用户 | 当目标读者、交付格式或高风险写操作不明确时，最多问 3 个短问题 |
| 注意完整性 | 每个步骤必须有明确输出物；缺少必要输入时追问而非假设 |

### 原则 2：description 是路由开关

description 决定 Skill 是否被加载。必须写清楚三件事：
1. **场景** — 用户说什么时应该触发
2. **输入** — 期望什么类型的输入
3. **输出** — 产出什么类型的结果

写 description 前先回答三个问题：
- 用户真实会怎么提出这个需求？
- 哪些相邻需求不应该触发？
- 加载这个 Skill 后，Agent 应该产出什么类型的结果？

如果三个问题答不清，先不要写正文。

| 差版本 | 好版本 |
|---|---|
| 帮助用户处理文档，提高写作效率 | 当用户提供飞书文档、Markdown 草稿或主题大纲，并要求整理成可发布的中文知识库文章时使用。输出结构化文档，包含标题、导语、章节、表格或 callout，并保留来源链接 |

### 原则 3：渐进式加载

Agent 不会一上来就读所有内容。Skill 的信息架构应分层：

| 层级 | 内容 | 加载时机 |
|---|---|---|
| **description**（必读） | 场景 + 输入 + 输出，≤ 200 字符 | 路由匹配阶段 |
| **SKILL.md 正文**（必读） | 触发条件、工作流、停止条件、输出要求 | 确认触发后 |
| **references/**（按需） | 长规范、API 文档、业务背景、模板 | 执行中需要时 |
| **scripts/**（按需） | 确定性动作的可执行脚本 | 对应步骤执行时 |
| **examples/**（按需） | 好结果、坏结果、边界样例 | 对照判断时 |
| **evals/**（按需） | 最小评测集 | 验证/迭代时 |

正文尽量控制在 500 行以内。长资料放 `references/`，按需读取。

### 原则 4：每个 Skill 都是一笔上下文税

- 新增 Skill 会增加 Agent 的路由判断负担
- 边界模糊、很少触发、没人维护的 Skill 会**污染路由、浪费上下文**
- **宁可漏触发也不要误触发** — 误触发的代价（跑错流程）远大于漏触发（退回普通对话）
- 未经 baseline 验证（3-5 个真实 case）的任务，不急着 Skill 化

### 原则 5：确定性动作脚本化，判断性动作留在 workflow

| 情况 | 做法 |
|---|---|
| 动作每次都一样，参数固定 | 写成 `scripts/` 里的可执行脚本 |
| 动作一样但参数随输入变化 | 在 workflow 里给出命令模板和参数说明 |
| 需要判断后选择不同路径 | 在 workflow 里写判断逻辑，每条路径给出对应命令 |
| 工具有复杂的失败模式 | 在 workflow 里列出常见错误和处理方式 |

过早脚本化会增加维护成本。只有当一个动作反复出现、参数稳定、且人工记忆容易出错时，才值得抽成脚本。

### 原则 6：可迁移性（跨平台、跨模型）

- 用意图描述行为，不绑定特定模型（不说"让某模型先做 X"）
- 用通用 CLI 接口，不依赖平台专属 plugin
- 流程中的判断写成"如果 X 则 Y"，不依赖特定模型的隐式能力
- Skill 目录路径写成平台自适应形式（见下方路径约定）
- 不硬编码模型名称或版本号

检验标准：把 Skill 复制到另一个 Agent 环境，不改 SKILL.md 就能跑通基本流程。

## 平台路径约定

不同 Agent 平台的 Skill 安装路径不同。生成和引用路径时，按以下规则自适应：

| 平台/环境 | Skill 根目录 | 示例 |
|---|---|---|
| TRAE | `.trae/skills/` | `.trae/skills/<skill-name>/SKILL.md` |
| Codex | `$CODEX_HOME/skills/` 或 `~/.codex/skills/` | `~/.codex/skills/<skill-name>/SKILL.md` |
| Qoder | `~/.qoderwork/skills/` | `~/.qoderwork/skills/<skill-name>/SKILL.md` |
| 通用约定 | `~/.agent/skills/` | `~/.agent/skills/<skill-name>/SKILL.md` |

**规则**：
- 生成 Skill 时，先询问用户的目标平台，按上表确定路径
- 如果用户未指定平台，默认用当前运行环境的路径
- Skill 内部引用自身文件时，用相对路径（如 `scripts/validate.sh`），不写绝对路径
- 如果平台有官方的 `init_skill` 或 `skill validate` 命令，优先使用官方工具

## 工作流程

### 第 1 步：判断是否值得 Skill 化

在动手写之前，先确认任务满足以下条件。**如果四个维度都不满足，建议用户先改 Prompt 或放进 Memory，不要创建 Skill。**

| 判断维度 | 适合 Skill 化 | 暂时别做 |
|---|---|---|
| 频率 | 每周都有，或同类需求反复出现 | 一次性脑暴，做完不会再出现 |
| 稳定性 | 流程大体固定，只是输入不同 | 每次都需重新定义目标和判断标准 |
| 交付物 | 能落到文档、代码、表格、报告等 | 只是开放式讨论，没有明确产出 |
| 风险 | 可以设置检查点和人工确认 | 高风险动作很多且无清晰审批边界 |

**Baseline 先行**：如果用户没有 3-5 个真实 case 的运行记录，建议先无 Skill 跑几遍，观察是否稳定暴露同类问题（总是漏步骤、总是用错工具、总是输出无法验收）。问题稳定复现时才值得 Skill 化。

### 第 2 步：确定 Skill 类型

| 类型 | 适合场景 | 重点写清楚 |
|---|---|---|
| **角色型** | 负责某一步判断（需求分析、代码 Review、文档编辑） | 判断框架、输入边界、输出标准 |
| **工具型** | 把 CLI 或工具用对（操作特定平台、处理表格、调 API） | 命令、参数、权限、异常处理 |
| **编排型** | 串起多个角色和工具（需求→设计→任务拆解） | 执行顺序、检查点、回退路径 |

三种类型可以组合，但不建议都塞进一个 Skill。边界越清楚越好维护。

### 第 3 步：引导用户编写 description

向用户收集以下信息：
1. 用户通常怎么描述这类任务？（收集 3-5 种常见说法做正例）
2. 输入是什么？（文件类型、链接、口头描述等）
3. 输出是什么？（文档、代码、报告等具体格式）
4. 哪些相邻任务**看起来像但不应该**触发？（写反例）

生成的 description 必须包含：**场景 + 输入 + 输出**，控制在 200 字符以内。

### 第 4 步：搭建 SKILL.md 骨架

按以下结构生成 SKILL.md：

```markdown
---
name: "<skill-name>"
description: "<场景 + 输入 + 输出，≤ 200 字符>"
---

# <Skill 标题>

## 触发条件
- 用户说 X、做 Y、提供 Z 时触发
- （写具体说法，不只写"用户想要 X"）

## 不适用
- 不处理 A 场景（解释为什么看起来像但不触发）
- 不做 B 动作

## 工作流程
1. 步骤一（具体动作 + 产出）
2. 步骤二（具体动作 + 产出）
3. ...

## 输出要求
- 格式规范（标题、章节、引用等具体格式）
- 质量标准（哪些必须有、哪些不能有）

## 停止条件
- 缺必要输入时：列出缺失项并追问，最多问 N 次
- 权限失败时：停止并报告原因
- 高风险动作：暂停等待人工确认
- 连续失败 N 次：停止并建议人工介入
- 事实不确定时：标注为假设或追问，不编造
```

### 第 5 步：填充工作流细节

将每个步骤写成具体动作：

- 用动词开头（读取、提炼、重组、校验、输出）
- 标注每一步的输出物
- 标注哪些步骤是"必须"、哪些是"可选"
- 标注哪些步骤需要调用工具/脚本
- 确定性动作给出完整命令模板和参数说明
- 给每个步骤标明失败时的回退路径

### 第 6 步：验收检查

生成完所有文件后，逐条检查：

- [ ] description 写了场景、输入、输出，用户换一种常见说法仍能触发
- [ ] 不适用场景已明确写出（有反例，不只写"适用于 Y"）
- [ ] 缺少必要输入时，Skill 知道该追问、假设还是停止
- [ ] 每个工作流步骤有明确动作，不只是原则性描述
- [ ] 输出格式固定，能被人或下一个流程继续使用
- [ ] 至少有 3 个标准样例、2 个边界样例、1 个反例
- [ ] 关键 CLI 或脚本调用有参数示例和失败处理
- [ ] 停止条件明确：缺资料、权限失败、高风险、连续失败有处理路径
- [ ] SKILL.md 正文不超过 500 行（长内容移入 references/）
- [ ] 没有硬编码模型名称（不出现具体模型名，即使作为反例也应尽量用通用表述）
- [ ] 工具调用用 CLI 或通用接口，非平台专属 plugin
- [ ] 流程中的判断写成"如果 X 则 Y"，不依赖特定模型隐式能力
- [ ] 文件路径使用平台自适应约定，不硬编码绝对路径

### 第 7 步：设计最小 Evals

生成 `evals/eval.json`，包含三类样本：

1. **标准样本**（3-5 个）：用户最常提交的真实任务
2. **边界样本**（2-3 个）：空输入、资料不全、格式混乱
3. **反例**（1-2 个）：不应触发的输入、之前误触发的案例

```json
{
  "skill_name": "<skill-name>",
  "evals": [
    {
      "id": 1,
      "type": "standard",
      "prompt": "<真实任务描述>",
      "expected_output": "<期望交付物>",
      "assertions": [
        { "id": "a1", "text": "<可检查的断言>" }
      ]
    },
    {
      "id": 2,
      "type": "boundary",
      "prompt": "<边界输入>",
      "expected_output": "<期望行为>",
      "assertions": [
        { "id": "b1", "text": "<边界情况的断言>" }
      ]
    },
    {
      "id": 3,
      "type": "anti",
      "prompt": "<不应触发的输入>",
      "expected_output": "不加载此 Skill",
      "assertions": [
        { "id": "c1", "text": "没有执行此 Skill 的流程" }
      ]
    }
  ]
}
```

### 第 8 步：创建配套文件

如果第 2 步确认需要脚本化，创建以下脚本：

- `scripts/init.sh` / `scripts/init.ps1` — 初始化 Skill 目录骨架
- `scripts/validate.sh` / `scripts/validate.ps1` — 校验 SKILL.md 格式和完整度

参考本 Skill 自带的 `scripts/` 目录。

## 迭代与维护指导

生成 Skill 后，**必须**向用户说明以下维护要点：

### SkillOpt 四步循环

| 步骤 | 动作 | 要点 |
|---|---|---|
| **Rollout**（执行） | 用当前 Skill 跑一批真实任务 | 积累 3-5 个执行结果（成功/失败/半通过） |
| **Reflect**（反思） | 对比成功和失败批次，找可复用规律 | 分析失败是触发问题、步骤遗漏还是工具误用 |
| **Edit**（编辑） | 对 Skill 做有限修改（add/delete/replace） | **每次只改 1-2 处**，不要一次性重写 |
| **Gate**（门控） | 在验证集上对比改前改后 | 用 evals 确认没有回归，不靠"感觉变好了" |

### Gotchas 飞轮

遇到真实失败时，**追加到 Skill 中**而非重写整份说明：

| 真实现象 | 优先补哪里 | 不要急着做什么 |
|---|---|---|
| 不该加载时加载了 | 补反例，收紧 description | 不要只在正文里写"谨慎使用" |
| 该加载时没加载 | 补真实 query 说法和正例 | 不要把 description 写成宽泛口号 |
| 流程某一步反复漏掉 | 写成必须动作或验收项 | 不要只写"注意完整性" |
| 工具或依赖经常漂移 | 补失败处理、版本检查或停止条件 | 不要把易变细节全塞进 SKILL.md |

### 多 Skill 路由冲突治理

- 每个 Skill 的 description 里明确写"不适用于 X 场景"（不只写"适用于 Y"）
- 新增 Skill 前，先确认相邻 Skill 的正例仍然通过
- 如果两个 Skill 有 **80% 步骤相同**，合并成一个、用条件分支区分
- 经验法则：**每新增一个 Skill，至少检查 2 个相邻 Skill 的 description 是否需要收紧**

### 退役条件

以下情况考虑合并、降级或删除：

1. 模型原生能力已覆盖该场景（定期跑"有 Skill vs 无 Skill"对比测试）
2. 维护成本超过收益（每次用都要手动修正，不如不用）
3. 流程已过时（底层工具或平台已换代）
4. 和新 Skill 大量重叠（80% 相同步骤）
5. 有 Skill vs 无 Skill 的表现差异消失（模型已内化这套做法）

退不退役不丢人。Skill 太多互相打架，Agent 反而更容易出错。

## 输出规范

### 目录结构

```
<skill-name>/
├── SKILL.md              # 必须 — 核心说明书（含 frontmatter）
├── README.md             # 推荐 — GitHub 项目说明（面向人类读者）
├── scripts/              # 可选 — 确定性脚本
│   ├── init.sh           #   Unix 初始化脚本
│   ├── init.ps1          #   Windows 初始化脚本
│   ├── validate.sh       #   Unix 校验脚本
│   └── validate.ps1      #   Windows 校验脚本
├── references/           # 可选 — 长规范、API 文档、业务背景（按需读取）
├── examples/             # 可选 — 好结果、坏结果、边界样例
├── evals/
│   └── eval.json         # 必须 — 最小评测集
└── LICENSE               # 推荐 — 开源许可证
```

### 生成步骤

1. 询问用户目标平台，确定 Skill 安装根目录
2. 创建 `<skill-name>/` 目录及所有子目录
3. 生成 `SKILL.md`（含 frontmatter 和完整正文）
4. 如有脚本需求，在 `scripts/` 下创建对应脚本
5. 生成 `evals/eval.json` 最小评测集
6. 向用户展示验收清单，逐条确认通过
7. 提醒用户：第一次用先跑 3-5 个 case，记录结果，攒够再优化
